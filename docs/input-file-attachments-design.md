# Input File Attachments (Design)

This document describes the design for letting workflow input steps exchange
files with the user through the `raymond serve` web interface — both files
uploaded by the user as part of an input response, and files written by the
workflow that the user can view or download alongside the input prompt.

It captures intent only. Implementation details (route registration, handler
internals, storage abstractions) are deliberately omitted.

For the surrounding context, see [daemon-server.md](daemon-server.md) and the
`<await>` semantics in [workflow-protocol.md](workflow-protocol.md).

## Motivation

Today, `<await>` input steps in `raymond serve` exchange only text. That is
sufficient for confirmations, free-form instructions, and short structured
answers, but it excludes a class of workflows that are otherwise well-suited
to the human-in-the-loop model:

- A workflow that asks the user to upload a document, image, or dataset for
  the agent to process.
- A workflow that produces an artifact (a generated PDF, a chart, a diff)
  and wants the user to inspect it before continuing.
- A workflow whose next step depends on multiple supporting files the user
  selects ad hoc.

The reference point for "files exposed via the dashboard" is `backlog_manager`
(`../backlog_manager`), which uses a hard-coded, hash-validated bundle upload
endpoint with a single global directory. Raymond's needs are different: files
are scoped to a specific input step within a specific run, the runtime should
not impose a fixed naming convention, and uploads should be a natural
extension of the existing input mechanism rather than a parallel system.

## Scope

In scope:

- A way for `<await>` to declare that an input step accepts file uploads
  and/or exposes files for the user to view or download.
- The HTTP surface, on-disk layout, and run-state record needed to support
  that declaration.
- The user experience for uploading, viewing, and downloading those files in
  the existing web UI.
- A workflow-author mechanism for the agent to learn which input step
  produced which files.

Out of scope (for this design):

- A general-purpose task file browser exposing arbitrary files in any task
  folder. File access is mediated by an input step.
- File exchange over the MCP transport. This design is HTTP-only; the MCP
  transport behavior is addressed only as a degradation note.
- Authentication or per-user access control. The daemon remains
  unauthenticated and assumes a trusted network, consistent with current
  behavior.
- External or object storage backends. Files live in the run's task folder
  on the daemon's local filesystem; their lifetime tracks that of the task
  folder.
- Mutation of files after upload (renaming, replacement, deletion via the
  UI).

## Core Concepts

### Per-input subdirectory

Files associated with an input step live in a subdirectory of the requesting
agent's task folder, dedicated to that input step:

```
<task folder>/inputs/<input_id>/
```

This subdirectory is the canonical, stable location for both directions of
file exchange once the await has been entered:

- **Uploads** from the user land here on submission.
- **Display files** are staged here by the runtime when the await is
  entered (see below), copied from wherever the workflow originally wrote
  them in the task folder.

Scoping by `input_id` means files belonging to different input steps within
the same run cannot collide, and the agent can reason about "files for input
X" by listing one well-defined directory.

`input_id` is generated by the runtime at await-entry time, not chosen by
the workflow author. The author therefore cannot write directly to
`inputs/<input_id>/` ahead of an await — the directory does not exist until
the await is reached. Display files are handled by staging (next section);
uploads do not have this problem because they only arrive after the await
is entered.

The `inputs/` segment is part of the runtime-managed layout. Workflow
authors should treat it as stable, read it for uploads in the state
following the await, and not write to it directly.

### Two modes: slot mode and bucket mode

An `<await>` that participates in file exchange opts into one of two modes:

**Slot mode.** The await declares specific named files. The UI shows a labeled
upload control per slot, and the server enforces the declared names. The
agent reads each file by its known path. Slot mode is appropriate when the
agent knows in advance what it is asking for (e.g. "upload your résumé as
`resume.pdf`").

**Bucket mode.** The await declares constraints only — for example, a maximum
file count, a MIME-type allowlist, and a maximum size per file or total. The
user chooses filenames. The agent scans the input subdirectory to discover
what arrived. Bucket mode is appropriate when the request is open-ended
("attach any supporting documents").

Both modes write to the same `<task folder>/inputs/<input_id>/` location, so
agent-side ergonomics are uniform; slot mode merely happens to know the
filenames in advance.

An await may also expose files for display alongside the prompt, regardless
of upload mode. Each display file is declared by a path relative to the
task folder (the source path the workflow wrote to) and an optional display
name. When the await is entered, the runtime copies each declared display
file from its source path into `inputs/<input_id>/` under the display name
(or its source basename if no display name is given). Missing source files
are reported as a startup-of-await error rather than a silent omission.

After staging, display files and uploads share the same physical directory
and the same URL space — the agent and the UI see a uniform view of "the
files for this input."

### The `{{input_id}}` template variable

For the agent to read uploads in bucket mode (or to reference files
relative to the input subdirectory in either mode), it needs the input
identifier. The state **immediately following** an input step receives an
additional template substitution, `{{input_id}}`, bound to the ID of the
input that resolved to reach that state. This is the only state in which
`{{input_id}}` is bound automatically.

Raymond does not have a general state-to-state context propagation
mechanism for arbitrary values. If a workflow needs `input_id` in a state
deeper than one transition past the await, the author plumbs it forward
explicitly through the transition `input` attribute, in the same manner
that `{{result}}` is carried today (for example, by passing a structured
payload that bundles the response value with the input id, or by writing
the input id to a file in the task folder during the immediately-following
state and reading it back later). Most workflows will not need this — the
state directly after the await is the natural place to consume uploads.

This is the only new template variable introduced by this design.

## The `<await>` Extension

The `<await>` tag gains optional file-related attributes on the opening tag
(no nested elements; the existing parser is regex-based and only sees opening-tag
attributes). The shipped attribute names are `upload_slots`, `upload_bucket`,
`upload_max_count`, `upload_max_size`, `upload_max_total_size`, `upload_mime`,
and `display_files`; see [workflow-protocol.md](workflow-protocol.md#file-attachments)
for the full attribute reference and examples.

Conceptually, an extended `<await>` may declare:

| Aspect | Slot mode | Bucket mode | Display only |
|--------|-----------|-------------|--------------|
| Named upload slots | required | n/a | n/a |
| Maximum file count | n/a | optional | n/a |
| MIME-type allowlist | optional (per slot) | optional | n/a |
| Maximum size per file | optional | optional | n/a |
| Files to display | optional | optional | required |

An await without any of these declarations behaves exactly as today
(text-only input). Adding them is opt-in and backwards-compatible with
existing workflows.

A single await may combine display files with either upload mode (e.g. show
the user a chart, then ask them to upload a corrected dataset).

## On-Disk Layout

```
<task folder>/
  inputs/
    <input_id_1>/
      <files for that input step>
    <input_id_2>/
      <files for that input step>
    ...
```

Properties:

- The `inputs/` directory and per-`input_id` subdirectories are created by
  the runtime when a file-bearing await is entered. Display files declared
  by the await are staged into the subdirectory at that point by copying
  from their source paths in the task folder; uploads are written into the
  subdirectory on submission.
- Files persist in place after the input resolves. They are not moved,
  renamed, or deleted by the runtime. Cleanup is the workflow author's
  responsibility (or happens implicitly when the task folder is removed).
- The path is stable and may be relied upon by workflow scripts. The
  `{{input_id}}` substitution exists so authors do not have to hard-code
  or reconstruct it in the state immediately following the await.
- The original display-file source paths in the task folder are not
  affected by staging — the runtime reads them, not the inverse. The
  workflow may overwrite or delete them after the await is entered without
  affecting the staged copies.

## HTTP API Surface

The existing input endpoint (`POST /runs/{id}/inputs/{input_id}`) extends to
accept `multipart/form-data` in addition to its current JSON body. Content
negotiation is by `Content-Type`:

- `application/json` — text-only response, current behavior preserved.
- `multipart/form-data` — text response (in a designated form field) plus
  zero or more file parts. Slot-mode parts are matched to declared slot
  names; bucket-mode parts may carry any name subject to constraints.

A new read endpoint serves files associated with a specific input:

- `GET /runs/{id}/inputs/{input_id}/files/<path>` — returns the file at
  `<path>` within that input's subdirectory.

A listing endpoint supports the UI rendering display files and resolved
upload records:

- `GET /runs/{id}/inputs/{input_id}/files` — returns metadata for files
  currently present in the input subdirectory (name, size, MIME type).

The pending-inputs endpoint extends to surface, for each pending input, the
file affordance the await declared (mode, slots, constraints, display files)
so the UI can render the appropriate form.

## Upload Semantics

**Validation.** The server enforces the await's declared constraints (slot
names in slot mode; count, size, MIME allowlist in bucket mode). Violations
return a 4xx with a structured error indicating which constraint failed.

**Filename normalization.** The server normalizes user-supplied filenames
before persistence: path separators, null bytes, control characters,
leading dots, and platform-reserved names are rejected. The server does
not silently rewrite a filename to a different one — if a name fails
normalization, the upload is rejected with a structured 4xx error so the
user can correct it.

**Collision handling.** Within a single multipart submission, two parts
with the same effective filename are a conflict. The UI prevents the
duplicate before sending where possible (filename already chosen in this
dialog), and the server returns `409 Conflict` if a duplicate is present
in the submission (e.g. via direct API use). Display files staged by the
runtime also count toward the namespace: an upload that would overwrite a
staged display file is a 409. The server does not auto-rename — surprise
suffixes confuse both the user and the agent.

Across input steps, collisions are impossible by construction because each
input has its own subdirectory. Concurrent submissions to the same input
are not a collision case: the first to validate resolves the input, and
the second is rejected as "input not pending" rather than 409.

**Atomicity.** A multipart submission is treated as a single transaction:
either all files validate and the input resolves, or none are persisted and
the input remains pending. Partial uploads do not leave files in the input
subdirectory.

**Size limits.** A server-enforced maximum applies to total submission
size and to per-file size. Uploads beyond the limits are rejected before
the input subdirectory is modified. The exact configuration mechanism is
left to implementation (see Deliberate Non-Decisions).

## Read / Serve Semantics

**Default disposition.** Files are served as
`Content-Disposition: attachment` by default. Clients may request inline
rendering by passing a query parameter (e.g. `?disposition=inline`) on
the read endpoint. The server honors that hint only for files whose
content type is on a vetted MIME-type allowlist; anything outside the
allowlist is served as an attachment regardless of the requested
disposition.

**Inline rendering.** The web UI uses the inline mechanism above to
render certain file types alongside the input prompt — at minimum,
common image MIME types and PDFs. The allowlist is server-defined and
not author-controllable; HTML, SVG, and other script-bearing types are
not on it.

**Content type determination.** The server determines content type by
sniffing file contents at the time the file is added to the input
subdirectory (on upload, or on staging for display files), and records
the result in the run-state catalog. The recorded type is what subsequent
listings and reads use; the server does not re-sniff at view time.

**Defense-in-depth headers.** Responses serving file content carry
`X-Content-Type-Options: nosniff` and a `Content-Security-Policy` consistent
with not executing arbitrary scripts from served files. HTML, SVG, and other
script-bearing types are not part of the inline allowlist.

**Path safety.** The `<path>` segment of the read endpoint is resolved
strictly within the input subdirectory; traversal attempts (`..`, absolute
paths, symlinks escaping the subdirectory) are rejected.

## Run State and History

The run state record gains, per resolved input:

- The text response (as today).
- A list of files associated with that input — name, size, MIME type, and a
  reference resolvable through the read endpoint.

This is what allows the run history view to re-display an input's full
context after it has resolved: the prompt, the files the user saw, the text
they returned, and the files they uploaded. Without this record, files
visible during the input would disappear from the UI the moment the input
resolved, even though they still exist on disk.

For display files, the record names the files that were staged at the
moment the await was entered, including their content types as determined
at staging time. For uploads, the record names the files that were
persisted as part of the response, including their content types as
determined at upload time. Once a file is in the catalog, its recorded
content type is fixed for the lifetime of the run-state record.

## Workflow Author Experience

A workflow that asks the user for a document looks roughly like:

1. A state authors a prompt that includes an `<await>` declaring the
   expected file affordance (slot mode with `resume.pdf`, or bucket mode
   with constraints).
2. When the await is entered, the runtime creates
   `<task folder>/inputs/<input_id>/` and stages any declared display
   files into it.
3. The user, in the web UI, sees the prompt with the appropriate upload
   controls and any display files rendered or linked.
4. On submission, uploaded files land in `<task folder>/inputs/<input_id>/`
   alongside the staged display files, the input resolves, and the
   workflow transitions to its next state.
5. The next state receives `{{input_id}}` and can read the uploaded files
   directly from the input subdirectory.

A workflow that wants the user to inspect an artifact before approving
follows the same shape: in some earlier state the agent writes the
artifact to a path within the task folder, and the `<await>` declares
that path as a display file source. The runtime stages it on entry; the
agent does not need to know `input_id` at the time it writes the
artifact.

## Interaction with the MCP Transport

The MCP transport does not have a natural representation for binary file
exchange in the input-resolution flow. Awaits that declare upload affordances
(slot or bucket mode) are not delivered via MCP elicitation: the await stays
pending, a warning is logged, and the user must complete it via the HTTP UI.

Awaits that declare only display files do degrade gracefully on MCP: the
elicitation prompt is augmented with absolute URLs (under the daemon's
configured base URL) for each staged file, so an MCP client can fetch them
out of band via the HTTP file content endpoint. If no base URL is configured
for the daemon, the URLs are omitted and the await effectively degrades to
text-only.

Workflow authors targeting both transports should design awaits that are
either text-only or text-plus-display, and avoid making upload affordances
mandatory for progress.

## Security Considerations

- The daemon remains unauthenticated. File ingress and egress inherit that
  posture: anyone who can reach the daemon can upload to and download from
  any pending or past input. Operators must continue to scope network
  exposure accordingly.
- Inline rendering is restricted to a vetted MIME-type allowlist. All other
  content is served as an attachment with `nosniff`.
- Path traversal in the read endpoint is rejected.
- Upload size and count limits are enforced at the server, not just the UI.
- The server does not execute, transform, or interpret uploaded files — it
  only persists them and serves them back.

## Deliberate Non-Decisions

The following are not specified by this document and are left for
follow-up design:

- Storage quotas across runs, retention policies, or archival.
- How to surface upload progress for very large files in the UI.
- Whether and how to support resumable uploads.
- Any future general-purpose file browser; this design intentionally does
  not preclude one but does not design for it either.

The following non-decisions were resolved during implementation:

- **`<await>` attribute syntax** — see
  [workflow-protocol.md](workflow-protocol.md#file-attachments) for the full
  attribute reference. Slots, bucket constraints, and display-file source
  paths ride on opening-tag attributes (`upload_slots`, `upload_bucket`,
  `upload_max_count`, `upload_max_size`, `upload_max_total_size`,
  `upload_mime`, `display_files`); no nested elements.
- **JSON shape for pending-inputs and file-listing endpoints** — see
  [daemon-server.md](daemon-server.md#file-bearing-awaits) for the response
  shapes.
- **Configuration mechanism for upload size limits** — server-wide caps are
  configured via `Server.SetDefaultUploadCaps(perFile, total, count)`,
  resolved in the precedence order: per-await bucket override > server
  config > built-in fallbacks (10 MiB per file, 100 MiB total, 10 files per
  submission). Slot-mode and display-only awaits inherit the server-wide
  caps; only bucket-mode `<await>` may override them per-await.
